# MSMC 全面优化与功能增强计划

## 一、项目现状分析

### 项目基本信息
- **项目名称**: McServerGuard (MSMC) - Minecraft 服务器管理客户端
- **技术栈**: WPF + .NET 9.0 + CommunityToolkit.Mvvm + MaterialDesignThemes
- **目标框架**: `net9.0-windows10.0.17763.0`
- **当前版本**: 0.1.0

### 核心模块结构
```
src/McServerGuard/
├── Models/              # 数据模型（ServerInstance, KnownServer, SystemMetrics等）
├── Services/
│   ├── ServerDetection/ # 服务器检测服务
│   ├── ConfigManagement/# 配置管理服务
│   ├── SystemMonitoring/# 系统监控服务
│   └── AIService/       # AI守护服务
├── ViewModels/          # MVVM 视图模型
└── Views/               # WPF 页面与控件
```

### 关键现有功能
1. **服务器检测**: 通过扫描 Java 进程识别 Minecraft 服务器（手动触发）
2. **配置编辑**: 支持 .properties/.yml/.json 格式配置文件的可视化编辑
3. **系统监控**: CPU/内存/磁盘/Java进程指标采集（2秒间隔）
4. **AI守护**: 崩溃预测、日志异常检测、配置优化
5. **JVM参数管理**: 预设参数、自定义参数、启动命令预览

## 二、任务分解与实施方案

### 任务1：服务器检测改为一秒一次的死循环

**现状**: 
- `ServerDetectionViewModel.DetectAsync()` 是手动触发的单次检测
- `MainViewModel` 启动时自动检测一次
- 检测通过 `IServerDetector.DetectAllAsync()` 完成全量扫描

**实施方案**:
1. 在 `IServerDetector` 接口中新增持续检测方法：`StartContinuousDetection(TimeSpan interval, Action<DetectionResult> callback, CancellationToken token)` 和 `StopContinuousDetection()`
2. 在 `ServerDetector` 中实现：使用 `while` 循环 + `Task.Delay(1000)` 的后台检测循环（而非 Timer，避免重入问题）
3. 在 `ServerDetectionViewModel` 中添加：
   - `IsAutoDetecting` 属性（自动检测状态）
   - 启动/停止自动检测的逻辑
   - 检测结果增量更新（而非全量替换，减少UI闪烁）
4. 在 `MainViewModel` 构造函数中启动自动检测循环
5. 应用退出时确保停止检测循环

**涉及文件**:
- `Services/ServerDetection/IServerDetector.cs` - 新增接口方法
- `Services/ServerDetection/ServerDetector.cs` - 实现循环检测
- `ViewModels/ServerDetectionViewModel.cs` - 添加自动检测状态与控制
- `ViewModels/MainViewModel.cs` - 启动自动检测
- `App.xaml.cs` - 退出时清理

**风险与注意**:
- 每秒全量扫描可能有性能开销，需要优化：进程扫描可轻量执行，配置文件扫描可缓存
- 检测循环必须可取消，避免退出时卡死
- UI 更新需在 Dispatcher 线程执行

### 任务4：强化API引用（兼容Windows 11）

**现状**:
- 目标框架已设为 `net9.0-windows10.0.17763.0`（Windows 10 1809+）
- 使用 `System.Management` 进行 WMI 查询
- 使用 `kernel32.dll` P/Invoke 获取内存信息
- 已引入 `Microsoft.Toolkit.Uwp.Notifications`（Win10+ 通知）

**实施方案**:
1. **升级 Windows SDK 版本**：
   - 将 `TargetFramework` 从 `net9.0-windows10.0.17763.0` 升级到 `net9.0-windows10.0.22000.0`（Windows 11 初始版本）
   - 或更高版本 `net9.0-windows10.0.22621.0`（Windows 11 22H2）

2. **强化 Win11 专属 API 调用**：
   - 添加 Win11 任务栏 API 支持（可选）
   - 确保 AppManifest 声明 Windows 11 兼容性
   - 添加 `app.manifest` 声明支持的操作系统

3. **Windows 11 通知优化**：
   - 检查 `Microsoft.Toolkit.Uwp.Notifications` 版本兼容性
   - 确保 Toast 通知在 Win11 上正常显示

4. **P/Invoke 兼容性加固**：
   - 确保 `kernel32.dll` 调用在 Win11 上稳定
   - 添加异常处理和降级策略

5. **添加应用程序清单 (app.manifest)**：
   - 声明 `supportedOS` 包含 Windows 11
   - 声明 DPI 感知（Per-Monitor V2）
   - 声明长路径支持

**涉及文件**:
- `McServerGuard.csproj` - 升级 TargetFramework
- 新增 `app.manifest` - Windows 兼容性声明
- `Services/SystemMonitoring/MemoryMonitor.cs` - 验证兼容性
- `Services/ToastNotificationService.cs` - 验证通知兼容

### 任务5：自提权与管理员权限检查

**现状**:
- 没有任何权限检查机制
- 应用以普通用户权限运行
- 某些操作（如读取其他进程命令行）可能需要管理员权限

**实施方案**:
1. **创建权限检查服务** `PrivilegeService`:
   - `IsRunningAsAdmin()` - 检查当前是否以管理员权限运行
   - `RequestElevation()` - 请求提升权限（重启自身并请求管理员权限）
   - `EnsureAdminPrivileges()` - 确保管理员权限，否则请求提权

2. **应用启动时权限检查**:
   - 在 `App.xaml.cs` 的 `OnStartup` 中检查权限
   - 如果不是管理员，显示提示对话框，询问是否提升权限
   - 用户选择"是"则重启程序并请求 UAC 提权
   - 用户选择"否"则继续以普通权限运行（但标记受限模式）

3. **受限模式提示**:
   - 在主窗口状态栏显示权限状态（"🔒 管理员模式"或"⚠️ 受限模式"）
   - 受限模式下某些功能可能禁用或警告

4. **添加 UAC 提权方法**:
   - 使用 `ProcessStartInfo.UseShellExecute = true` + `Verb = "runas"` 重启自身提权

5. **app.manifest 设置**:
   - 设置 `requestedExecutionLevel level="requireAdministrator"` 可选
   - 或者保持 `asInvoker`，在运行时按需提权

**涉及文件**:
- 新增 `Services/PrivilegeService.cs` - 权限检查与提权服务
- `App.xaml.cs` - 启动时权限检查
- `app.manifest` - 执行级别声明
- `ViewModels/MainViewModel.cs` - 显示权限状态
- `Views/MainWindow.xaml` - 状态栏权限显示

**注意**: 默认采用按需提权模式（不强制管理员），因为大多数功能不需要管理员权限，只有读取其他进程命令行等少数操作需要。可以在执行检测时检查权限并自动尝试提权。

### 任务6：优化多线程和渲染管线 + 自动内存释放

**现状**:
- 使用 `Dispatcher.Invoke` 进行 UI 线程调度
- 配置加载使用分批动画入场
- 没有显式的内存管理和资源释放机制
- 系统监控使用 `Timer` 定时采集

**实施方案**:
1. **多线程优化**:
   - 统一使用 `ConfigureAwait(false)` 减少 UI 线程阻塞（非UI操作）
   - 检测循环使用独立的后台线程，避免占用 ThreadPool
   - 使用 `Channel<T>` 或 `AsyncQueue` 进行线程间通信，替代直接 Dispatcher.Invoke
   - 服务器检测使用轻量级的进程状态轮询（每秒检查进程是否还在），全量扫描可降低频率

2. **渲染管线优化**:
   - 启用 WPF 硬件加速（默认已启用，但确保配置正确）
   - 减少 `ObservableCollection` 频繁变更导致的布局更新
   - 对列表控件启用虚拟化（`VirtualizingStackPanel`），特别是配置项列表
   - 合并动画，减少同时触发的 Storyboard 数量
   - 使用 `UIElement.CacheMode` 缓存复杂可视化元素
   - 优化颜色刷资源，减少动态资源查找开销

3. **内存自动释放**:
   - 添加 `MemoryPressureService` 监控应用自身内存
   - 当内存占用超过阈值时触发 GC 回收
   - 页面切换时清理前一页的大对象（如图像、大集合）
   - 实现 `IDisposable` 模式，确保服务正确释放
   - 历史数据（监控历史）设置上限，超出自动清理
   - 检测结果缓存定期清理

4. **资源管理**:
   - `PerformanceCounter` 等非托管资源确保正确 Dispose
   - WMI 查询对象及时释放
   - 文件流确保 using 模式

**涉及文件**:
- `Services/ServerDetection/ServerDetector.cs` - 检测循环优化
- `Services/SystemMonitoring/SystemMonitor.cs` - 采集优化
- `ViewModels/ConfigEditorViewModel.cs` - 内存优化
- `ViewModels/SystemMonitorViewModel.cs` - 历史数据管理
- `App.xaml.cs` - 渲染优化配置、内存监控
- 新增 `Services/MemoryOptimizationService.cs` - 内存优化服务（可选）
- `Views/ConfigEditorPage.xaml` - 启用虚拟化

### 任务7：修复系统监控 - 内存占用识别

**现状**:
- `MemoryMonitor` 使用 `GlobalMemoryStatusEx` (kernel32.dll) 获取内存
- `SystemMonitor.CollectSnapshot()` 采集内存信息
- 内存使用率 = (总内存 - 可用物理内存) / 总内存
- 可能存在的问题：
  - 可用内存计算不准确（没有考虑 Standby 列表等）
  - 进程内存统计可能有偏差
  - Java 进程内存识别可能不准确

**实施方案**:
1. **内存监控修复**:
   - 修正 `GetUsedMemory()` 计算方式：使用 `ullTotalPhys - ullAvailPhys` 是正确的，但需要确认 `ullAvailPhys` 含义
   - 添加缓存机制，避免每秒都做 P/Invoke（虽然开销不大）
   - 增加对 Windows 11 新内存管理特性的兼容

2. **Java 进程内存统计修复**:
   - 当前使用 `Process.WorkingSet64` 统计 Java 进程内存
   - 补充 `PrivateMemorySize64`（私有内存）和 `VirtualMemorySize64`（虚拟内存）
   - 修复进程枚举时的资源泄漏（确保每个 Process 对象被正确 Dispose）
   - 处理进程在枚举和读取之间退出的竞态条件

3. **内存信息展示增强**:
   - 区分"物理内存使用率"和"提交内存使用率"
   - 显示可用内存而非只显示使用率
   - 增加内存压力指示（高/中/低）

4. **SystemMetrics 模型扩展**（可选）:
   - 增加 `AvailableMemoryBytes` 字段
   - 增加 `JavaPrivateBytes` 字段
   - 增加 `CommitChargeBytes` 字段

5. **修复验证**:
   - 对比任务管理器的数据，确认数值一致
   - 处理权限不足时的降级显示

**涉及文件**:
- `Services/SystemMonitoring/MemoryMonitor.cs` - 内存计算修复
- `Services/SystemMonitoring/SystemMonitor.cs` - Java 进程统计修复
- `Models/SystemMetrics.cs` - 扩展字段（可选）
- `ViewModels/SystemMonitorViewModel.cs` - 展示增强
- `Views/SystemMonitorPage.xaml` - UI 展示更新

## 三、实施顺序建议

```
优先级从高到低：

1. 任务5（权限检查）- 基础安全性，先做避免后续功能受权限影响
2. 任务7（内存监控修复）- 修复现有Bug
3. 任务1 + 任务2（自动检测 + 刷新按钮）- 关联紧密，一起做
4. 任务4（Win11 兼容）- API 层面的兼容性
5. 任务3（配置功能增强）- 功能增强，工作量较大
6. 任务6（性能优化）- 全面优化，放在最后验证效果
```

## 四、潜在风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 每秒全量扫描导致 CPU 占用过高 | 性能问题 | 优化：进程存在性检查每秒做，全量扫描每10秒做 |
| UAC 提权导致用户体验下降 | 用户体验 | 默认普通模式运行，需要时提示提权而非强制 |
| Win11 API 升级导致 Win10 不兼容 | 兼容性 | 保留最低版本支持，使用版本检测调用新API |
| 自动检测循环导致内存泄漏 | 稳定性 | 严格管理 CancellationToken，退出时确保清理 |
| 配置编辑多目录扫描性能差 | 用户体验 | 异步扫描 + 增量加载 + 缓存结果 |
| WPF 虚拟化导致数据绑定问题 | UI  Bug | 充分测试，必要时回退虚拟化 |

## 五、验收标准

1. **自动检测**: 应用启动后自动开始检测，服务器列表每秒更新，检测状态正常显示
2. **刷新按钮**: UI 显示"刷新"按钮和刷新图标，点击后立即执行一次检测
3. **配置功能增强**: 配置页顶部有服务器选择下拉，选择后自动扫描该目录的所有配置文件，文件列表按目录分组显示
4. **Win11 兼容**: 在 Windows 11 上所有功能正常运行，通知、内存检测等正常工作
5. **权限检查**: 启动时检测权限，非管理员时显示提示，可一键提权重启
6. **性能优化**: 内存占用稳定，长时间运行无明显增长，UI 响应流畅
7. **内存监控修复**: 内存数据与任务管理器数据一致，Java 进程统计准确
