# 强化管理员权限检测与进程扫描可见性

## 一、问题诊断

用户反馈"显示管理员权限然而进程扫不到"。经全面排查，存在两层根因：

### 根因 1：权限检测方法不够可靠
- [PrivilegeService.cs:103-116](file:///workspace/src/McServerGuard/Services/PrivilegeService.cs) 用 `WindowsPrincipal.IsInRole(WindowsBuiltInRole.Administrator)` 检测
- 该方法在 UAC 标准场景下有效，但在 `dotnet run` 开发场景或 UAC 边界设置下可能误报
- 未使用更可靠的 `TokenElevation` P/Invoke（直接查询进程令牌是否被提升）

### 根因 2：进程扫描失败被静默吞掉（最直接的"扫不到"原因）
- [ProcessScanner.cs:255-293](file:///workspace/src/McServerGuard/Services/ServerDetection/ProcessScanner.cs) 的 `GetCommandLine` 方法：
  - WMI 查询 `Win32_Process.CommandLine` 失败时（`UnauthorizedAccessException`/`COMException`），只打 `Log.Debug`，返回 `string.Empty`
  - 调用方 [ProcessScanner.cs:70-74](file:///workspace/src/McServerGuard/Services/ServerDetection/ProcessScanner.cs#L70-L74) 收到空字符串后直接 `continue` 跳过该 Java 进程
  - **UI 完全无反馈**——用户看到的是"扫不到"，但实际是"WMI 查询失败被跳过"
- 同样的静默吞异常模式存在于 `IsProcessLaunchedByShell`（L195-243）

### 根因 3：manifest 与运行时提权逻辑矛盾
- `app.manifest` 配置 `requireAdministrator`，正常发布时进程必然 elevated
- 但 `App.xaml.cs` 仍有运行时提权提示逻辑——在 `dotnet run` 场景下 manifest 不生效，导致矛盾

## 三、具体改动方案

### 改动 1：PrivilegeService 改用 TokenElevation P/Invoke

**文件**：`PrivilegeService.cs`，`CheckIsAdmin` 方法（L103-116）

**问题**：`IsInRole(Administrator)` 在边界场景可能误报。

**方案**：新增 P/Invoke 声明，用 `OpenProcessToken` + `GetTokenInformation(TokenElevation)` 检测进程令牌是否真正被提升。保留 `IsInRole` 作为回退：

```csharp
// 新增 P/Invoke 声明（放在 PrivilegeService 类内部或底部）
[DllImport("advapi32.dll", SetLastError = true)]
[return: MarshalAs(UnmanagedType.Bool)]
private static extern bool OpenProcessToken(IntPtr ProcessHandle, uint DesiredAccess, out IntPtr TokenHandle);

[DllImport("advapi32.dll", SetLastError = true)]
[return: MarshalAs(UnmanagedType.Bool)]
private static extern bool GetTokenInformation(IntPtr TokenHandle, int TokenInformationClass,
    out int TokenInformation, int TokenInformationLength, out int ReturnLength);

[DllImport("kernel32.dll", SetLastError = true)]
[return: MarshalAs(UnmanagedType.Bool)]
private static extern bool CloseHandle(IntPtr hObject);

[DllImport("kernel32.dll")]
private static extern IntPtr GetCurrentProcess();

private const uint TOKEN_QUERY = 0x0008;
private const int TokenElevation = 20;  // TOKEN_INFORMATION_CLASS.TokenElevation

/// <summary>
/// 用 TokenElevation 检测进程令牌是否真正被提升（比 IsInRole 更可靠）
/// </summary>
private static bool IsProcessElevated()
{
    try
    {
        if (!OpenProcessToken(GetCurrentProcess(), TOKEN_QUERY, out var tokenHandle))
            return false;
        try
        {
            if (GetTokenInformation(tokenHandle, TokenElevation, out var elevation,
                sizeof(int), out _))
            {
                return elevation != 0;
            }
            return false;
        }
        finally
        {
            CloseHandle(tokenHandle);
        }
    }
    catch
    {
        return false;
    }
}
```

修改 `CheckIsAdmin` 优先用 `IsProcessElevated`，失败时回退到 `IsInRole`：

```csharp
private static bool CheckIsAdmin()
{
    try
    {
        // 优先用 TokenElevation 检测真实提权状态（最可靠）
        if (IsProcessElevated()) return true;

        // 回退到 IsInRole（兼容非 Windows 或 P/Invoke 失败场景）
        using var identity = WindowsIdentity.GetCurrent();
        var principal = new WindowsPrincipal(identity);
        return principal.IsInRole(WindowsBuiltInRole.Administrator);
    }
    catch (Exception ex)
    {
        Log.Warning(ex, "⚠️ 检查管理员权限失败");
        return false;
    }
}
```

**2b. GetCommandLine 失败时提升日志级别 + 记录跳过**

修改 `GetCommandLine` 方法（L255-293），把 `UnauthorizedAccessException` 和 `COMException` 从 `Log.Debug` 提升到 `Log.Warning`，并设置跳过原因：

```csharp
private string GetCommandLine(int processId)
{
    Log.Debug("🔧 获取命令行: PID={Pid}", processId);
    try
    {
        using var searcher = new ManagementObjectSearcher(
            $"SELECT CommandLine FROM Win32_Process WHERE ProcessId = {processId}");
        using var collection = searcher.Get();
        foreach (var obj in collection)
        {
            var cmdLine = obj["CommandLine"]?.ToString();
            if (!string.IsNullOrWhiteSpace(cmdLine))
            {
                // ... 原有转义和日志逻辑
                return cmdLine;
            }
        }
        // WMI 查询成功但 CommandLine 为空（可能是系统进程或权限边界）
        Log.Warning("⚠️ PID={Pid} 的 CommandLine 为空（可能是跨用户/服务进程）", processId);
    }
    catch (System.Runtime.InteropServices.COMException ex)
    {
        Log.Warning(ex, "⚠️ WMI 查询失败（COM 异常）PID={Pid}: {Message}", processId, ex.Message);
        LastSkipReason = "WMI 服务异常";
    }
    catch (UnauthorizedAccessException ex)
    {
        Log.Warning(ex, "⚠️ WMI 查询失败（权限不足）PID={Pid}: {Message}", processId, ex.Message);
        LastSkipReason = "权限不足，无法读取进程命令行";
    }
    catch (Exception ex)
    {
        Log.Warning(ex, "⚠️ 获取命令行失败 PID={Pid}: {Message}", processId, ex.Message);
        LastSkipReason = ex.Message;
    }

    return string.Empty;
}
```

**2c. ScanServerProcesses 中记录跳过计数**

修改 L70-74 的跳过逻辑：

### 改动 3：MainViewModel 暴露扫描跳过信息

**文件**：`MainViewModel.cs`

在 ViewModel 中新增属性，让 UI 能显示扫描跳过情况。需要从 ServerDetector 或 ProcessScanner 获取 `LastSkippedCount`。

```csharp
/// <summary>
/// 进程扫描跳过计数文本（用于状态栏提示）
/// </summary>
public string ScanSkipWarning => /* 从 ServerDetector 获取 LastSkippedCount */ 
    _processScanner.LastSkippedCount > 0 
        ? $"⚠️ 已跳过 {_processScanner.LastSkippedCount} 个无法访问的进程" 
        : string.Empty;
```

> 注意：需要确认 `ProcessScanner` 是否已注入到 `MainViewModel` 或可通过 `ServerDetector` 访问。如果访问链路过长，可在 `ServerDetector` 上暴露一个 `LastSkippedCount` 透传属性。

## 四、不改动项

- **app.manifest**：保留 `requireAdministrator`，这是正确的强制提权配置
- **App.xaml.cs 运行时提权逻辑**：虽然与 manifest 矛盾，但作为 `dotnet run` 场景的回退提示，删除会影响开发体验。本次不改动
- **AdminPrivilegeService**：孤儿服务，仅 `SystemMonitor` 用了 `GetWindowsVersion()`，不影响本次问题
- **PrivilegeChanged 事件**：保留声明但不强制实现，当前场景不需要运行时刷新

## 五、验证步骤

1. **编译验证**：`dotnet build` 无错误
2. **权限检测验证**：
   - 以管理员身份运行：`IsRunningAsAdmin` 返回 `true`，UI 显示"管理员模式"
   - `dotnet run` 场景（未提权）：`IsProcessElevated` 返回 `false`，UI 显示"受限模式"
3. **进程扫描验证**：
