# 服务器检测与系统监控全面修复计划

## 一、问题分析与研究结论

### 1.1 服务器检测循环问题
**现状分析**：
- 当前代码中服务器检测 (`DetectAsync`) 是手动触发的，启动时自动执行一次
- 系统监控 (`SystemMonitor`) 使用 `System.Threading.Timer` 每2秒采集一次
- 状态栏时钟每秒更新一次（仅UI显示，不影响性能）

**用户描述的"1秒死循环"可能原因**：
- 可能是用户观察到的系统监控频繁采样
- 或者是某个未被发现的后台循环
- 需要添加可配置的自动检测功能，替代硬编码循环

### 1.2 内存分析一直为0的问题
**根本原因分析**：
- `MemoryMonitor` 使用 P/Invoke 调用 `GlobalMemoryStatusEx`
- 可能的问题点：
  1. P/Invoke 声明在 Win11 上可能有兼容性问题
  2. 结构体 `MEMORYSTATUSEX` 的 `dwLength` 初始化可能有问题
  3. 非管理员权限下部分 WMI 查询失败
  4. `SystemMonitor.StartMonitoring` 的 Timer 回调可能有异常被静默吞掉

### 1.3 Win11 运行兼容问题
**潜在问题点**：
- WMI 查询在 Win11 上可能需要更高权限
- 某些系统 API 行为变化
- `System.Management` 包版本兼容性
- 进程命令行获取可能受 UAC 影响

### 1.4 软件自提权/管理员权限提醒
**现状**：
- 当前无任何管理员权限检查
- 部分功能（WMI 查询、进程信息获取）需要管理员权限才能完整工作

### 1.5 软件内启动服务器无法获取PID和内存
**根本原因**：
- `ServerManagerService.StartServer` 启动的是 `cmd.exe`（通过 `/c` 参数执行 java 命令）
- 返回的 `Process` 对象是 cmd.exe 的，PID 是 cmd 的 PID
- 真正的 `java.exe` 是 cmd.exe 的子进程
- 后续检测时通过命令行匹配 JAR 文件名来查找，但存在延迟和匹配失败的可能

## 三、详细修改步骤

### 步骤1：创建管理员权限服务
**目标**：实现管理员权限检查和UAC提权功能

**修改内容**：
1. 创建 `AdminPrivilegeService` 类
   - `IsRunningAsAdmin()`: 检查当前是否以管理员身份运行
   - `RestartAsAdmin()`: 以管理员身份重启程序
   - `CheckAndPromptAdmin()`: 检查权限，如需则提醒用户

### 步骤2：修复内存监控（内存分析为0问题）
**目标**：解决内存数据一直为0的问题，增强Win11兼容性

**修改内容**：
1. 修复 `MemoryMonitor.cs` 中的 P/Invoke 声明
   - 确保 `MEMORYSTATUSEX` 结构体布局正确
   - 确保 `dwLength` 在调用前正确初始化
   
2. 添加备用采集方案
   - 方案A：使用 `PerformanceCounter` ("Memory", "Available MBytes")
   - 方案B：使用 WMI 查询 `Win32_OperatingSystem` 的 `TotalVisibleMemorySize` / `FreePhysicalMemory`
   - 主方案失败时自动降级到备用方案

3. 增强错误日志
   - 记录失败的具体原因（返回值、错误码）
   - 记录 Windows 版本信息

### 步骤3：修复服务器启动PID跟踪问题
**目标**：软件内启动服务器后能正确获取java进程的PID和内存占用

**修改内容**：
1. 修改 `ServerManagerService.StartServer`
   - 方案：不再通过 cmd.exe 启动，直接启动 java.exe
   - 保留 cmd 窗口方式作为可选配置
   - 直接启动 java.exe 可以直接获得正确的 PID

2. 创建 `ServerProcessTracker` 服务
   - 跟踪软件启动的服务器进程
   - 缓存 Process 对象，定期更新内存占用
   - 进程退出时自动清理

3. 增强 `ServerInstance` 模型
   - 添加 `MemoryUsageBytes` 属性
   - 添加 `CpuUsagePercent` 属性

### 步骤4：修复服务器检测循环问题
**目标**：用可控的、可配置的自动检测替代硬编码死循环

**修改内容**：
1. 在 `ServerDetectionViewModel` 中添加自动检测功能
   - 添加 `IsAutoDetectEnabled` 属性（开关）
   - 添加 `AutoDetectIntervalSeconds` 属性（可配置间隔，默认30秒）
   - 使用 `DispatcherTimer` 实现定时检测
   - 检测页面不可见时暂停，节省资源

2. 添加配置持久化
   - 自动检测开关和间隔保存到配置中

### 步骤5：增强Win11兼容性
**目标**：确保软件在Win11上稳定运行

**修改内容**：
1. 添加 Windows 版本检测
   - 识别 Win11 并启用特定兼容性逻辑

2. 优化 WMI 查询
   - 所有 WMI 查询添加超时设置
   - 失败时提供降级方案
   - 减少不必要的 WMI 调用频率

3. 进程命令行获取优化
   - 优先使用 .NET 自带 API
   - WMI 作为备选
   - 添加缓存机制减少重复查询

### 步骤6：集成权限提醒到UI
**目标**：用户启动时如果不是管理员，给出明确提醒

**修改内容**：
1. 在 `App.xaml.cs` 启动时检查权限
   - 非管理员模式下记录警告日志
   
2. 在 `MainViewModel` 中添加权限状态
   - `IsRunningAsAdmin` 属性
   - 启动时显示提示（Snackbar）

3. 在设置页面添加"以管理员身份重启"按钮
   - 点击后触发 UAC 提权重启

## 四、潜在依赖和注意事项

### 4.1 技术依赖
- `System.Management` 包已在项目中引用（用于 WMI）
- UAC 提权使用 `ProcessStartInfo.Verb = "runas"`
- 不需要额外新增 NuGet 包

### 4.2 兼容性考虑
- 直接启动 java.exe vs 通过 cmd.exe 启动的区别：
  - 直接启动：没有 cmd 窗口，无法看到控制台输出
  - cmd 启动：有控制台窗口，但 PID 跟踪麻烦
  - **建议**：默认直接启动 java，添加"显示控制台窗口"选项

### 4.3 性能影响
- 自动检测间隔设置为30秒（默认），避免过度消耗资源
- WMI 查询添加缓存，减少重复调用
- 进程内存更新间隔与系统监控同步（2秒）

### 4.4 风险点
1. **直接启动 java.exe 可能导致用户无法看到控制台输出**
   - 缓解：保留 cmd 模式作为选项，默认直接启动
   
2. **UAC 提权重启可能丢失当前状态**
   - 缓解：提权前提示用户保存工作

3. **WMI 在某些系统上可能响应很慢**
   - 缓解：设置超时，提供降级方案
