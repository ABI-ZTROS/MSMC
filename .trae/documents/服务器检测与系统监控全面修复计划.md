# 服务器检测与系统监控全面修复计划

## 一、问题分析与研究结论

### 1.1 服务器检测1秒死循环问题
**现状**：
- 服务器检测被硬编码为1秒执行一次的死循环
- "开始检测"按钮启动后会一直循环检测
- 持续消耗系统资源，影响性能

**修复目标**：
- 移除死循环，改为单次检测
- "开始检测"按钮改为"刷新"按钮
- 点击后只执行一次检测，不会持续循环
- 如需要自动检测，改为可配置、可开关的定时检测（默认关闭）

### 1.2 内存分析一直为0的问题
**根本原因分析**：
- `MemoryMonitor` 使用 P/Invoke 调用 `GlobalMemoryStatusEx`
- 可能的问题点：
  1. P/Invoke 声明在 Win11 上可能有兼容性问题
  2. 结构体 `MEMORYSTATUSEX` 的 `dwLength` 初始化可能有问题
  3. 非管理员权限下部分 WMI 查询失败
  4. `SystemMonitor.StartMonitoring` 的 Timer 回调可能有异常被静默吞掉

### 1.3 Win11 运行兼容问题
**潜在问题点**：
- WMI 查询在 Win11 上可能需要更高权限
- 某些系统 API 行为变化
- `System.Management` 包版本兼容性
- 进程命令行获取可能受 UAC 影响

### 1.4 软件自提权/管理员权限提醒
**现状**：
- 当前无任何管理员权限检查
- 部分功能（WMI 查询、进程信息获取）需要管理员权限才能完整工作

### 1.5 软件内启动服务器无法获取PID和内存
**根本原因**：
- `ServerManagerService.StartServer` 启动的是 `cmd.exe`（通过 `/c` 参数执行 java 命令）
- 返回的 `Process` 对象是 cmd.exe 的，PID 是 cmd 的 PID
- 真正的 `java.exe` 是 cmd.exe 的子进程
- 后续检测时通过命令行匹配 JAR 文件名来查找，但存在延迟和匹配失败的可能

## 三、详细修改步骤

### 步骤1：修复服务器检测1秒死循环
**目标**：移除硬编码的1秒死循环，改为单次刷新检测

**修改内容**：
1. 查找并移除服务器检测的死循环代码（1秒间隔的while循环/定时器）
2. `DetectAsync` 保持单次执行一次检测就结束，不再循环
3. UI 按钮文字从"开始检测"改为"刷新"
4. 按钮图标从放大镜改为刷新图标
5. ToolTip 从"扫描运行中的服务器进程"改为"刷新服务器列表"
6. 添加可选的自动检测功能（默认关闭，可配置间隔）
   - 使用 `DispatcherTimer` 实现
   - 可在设置中开关和调整间隔
   - 检测页面不可见时暂停

### 步骤2：创建管理员权限服务
**目标**：实现管理员权限检查和UAC提权功能

**修改内容**：
1. 创建 `AdminPrivilegeService` 类
   - `IsRunningAsAdmin()`: 检查当前是否以管理员身份运行
   - `RestartAsAdmin()`: 以管理员身份重启程序
   - `CheckAndPromptAdmin()`: 检查权限，如需则提醒用户

### 步骤3：修复内存监控（内存分析为0问题）
**目标**：解决内存数据一直为0的问题，增强Win11兼容性

**修改内容**：
1. 修复 `MemoryMonitor.cs` 中的 P/Invoke 声明
   - 确保 `MEMORYSTATUSEX` 结构体布局正确
   - 确保 `dwLength` 在调用前正确初始化
   
2. 添加备用采集方案
   - 方案A：使用 `PerformanceCounter` ("Memory", "Available MBytes")
   - 方案B：使用 WMI 查询 `Win32_OperatingSystem` 的 `TotalVisibleMemorySize` / `FreePhysicalMemory`
   - 主方案失败时自动降级到备用方案

3. 增强错误日志
   - 记录失败的具体原因（返回值、错误码）
   - 记录 Windows 版本信息

### 步骤4：修复服务器启动PID跟踪问题
**目标**：软件内启动服务器后能正确获取java进程的PID和内存占用

**修改内容**：
1. 修改 `ServerManagerService.StartServer`
   - 默认直接启动 java.exe，不通过 cmd.exe 中转
   - 直接获得正确的 java 进程 PID
   - 保留"显示控制台窗口"选项（勾选时才用 cmd 方式启动）

2. 增强进程跟踪
   - 启动后保存 Process 对象引用
   - 可通过 Process 对象直接获取内存占用
   - 进程退出时自动清理

3. 增强 `ServerInstance` 模型
   - 添加 `MemoryUsageBytes` 属性（运行时内存占用）
   - 添加 `CpuUsagePercent` 属性

### 步骤5：增强Win11兼容性
**目标**：确保软件在Win11上稳定运行

**修改内容**：
1. 添加 Windows 版本检测
   - 识别 Win11 并启用特定兼容性逻辑

2. 优化 WMI 查询
   - 所有 WMI 查询添加超时设置
   - 失败时提供降级方案
   - 减少不必要的 WMI 调用频率

3. 进程命令行获取优化
   - 优先使用 .NET 自带 API
   - WMI 作为备选
   - 添加缓存机制减少重复查询

### 步骤6：集成权限提醒到UI
**目标**：用户启动时如果不是管理员，给出明确提醒

**修改内容**：
1. 在 `App.xaml.cs` 启动时检查权限
   - 非管理员模式下记录警告日志
   
2. 在 `MainViewModel` 中添加权限状态
   - `IsRunningAsAdmin` 属性
   - 启动时显示提示（Snackbar）

3. 在设置页面添加"以管理员身份重启"按钮
   - 点击后触发 UAC 提权重启

## 四、潜在依赖和注意事项

### 4.1 技术依赖
- `System.Management` 包已在项目中引用（用于 WMI）
- UAC 提权使用 `ProcessStartInfo.Verb = "runas"`
- 不需要额外新增 NuGet 包

### 4.2 兼容性考虑
- 直接启动 java.exe vs 通过 cmd.exe 启动的区别：
  - 直接启动：没有 cmd 窗口，无法看到控制台输出
  - cmd 启动：有控制台窗口，但 PID 跟踪麻烦
  - **方案**：默认直接启动 java，添加"显示控制台窗口"选项

### 4.3 性能影响
- 移除1秒死循环后，CPU 占用会显著降低
- 可选自动检测默认关闭，用户按需开启
- WMI 查询添加缓存，减少重复调用

### 4.4 风险点
1. **直接启动 java.exe 可能导致用户无法看到控制台输出**
   - 缓解：保留 cmd 模式作为选项，默认直接启动
   
2. **UAC 提权重启可能丢失当前状态**
   - 缓解：提权前提示用户保存工作

3. **WMI 在某些系统上可能响应很慢**
   - 缓解：设置超时，提供降级方案

## 五、验证方案

### 5.1 死循环修复验证
1. 启动软件，点击"刷新"按钮
2. 确认只执行一次检测
3. 确认检测完成后按钮恢复可用状态
4. 观察任务管理器，确认 CPU 占用正常

### 5.2 内存分析修复验证
1. 在 Win11 上启动软件
2. 进入系统监控页面
3. 确认内存数据正常显示（非0）
4. 检查日志中无内存监控错误

### 5.3 PID跟踪验证
1. 通过软件启动一个测试服务器
2. 确认服务器列表中显示正确的 PID（是 java 进程的 PID，不是 cmd 的）
3. 确认内存占用数据正常更新
4. 停止服务器后确认状态更新

### 5.4 管理员权限验证
1. 以普通用户身份启动软件
2. 确认收到非管理员提醒
3. 点击"以管理员身份重启"
4. 确认 UAC 弹窗出现
5. 确认重启后软件以管理员身份运行

### 5.5 Win11兼容性验证
1. 在 Win11 系统上完整测试所有功能
2. 确认无崩溃、无异常
3. 确认 WMI 查询正常工作
4. 确认进程信息正确获取
